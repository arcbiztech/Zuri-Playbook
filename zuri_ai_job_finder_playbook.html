What This Agent Does

This is a production-grade AI agent built to automate an entire job search pipeline — from discovery to a tailored, application-ready document — with zero manual searching required.

It runs on a schedule, finds real job listings from targeted sources, scores each one across seven dimensions aligned to a specific candidate profile, and generates customized resumes and cover letters only for high-scoring matches.

This playbook documents the agent's architecture, logic, decision rules, and team touchpoints — written so that any cross-functional team member can understand, configure, and build on it.

Human in the Loop

The AI surfaces and scores opportunities. Every application decision remains with the human. The agent advises — it does not decide.

Config-Driven, Not Hard-Coded

All parameters — keywords, salary floor, scoring weights, job boards — live in an Airtable Config table. Non-technical team members can update the system by editing a single cell. No automation editing required.

Modular by Design

Each scenario has exactly one job. A failure in sourcing does not break scoring. A scoring delay does not block document generation. Built like professional software.

Why This Approach

Why an Agent-Based Approach Matters for Teams

Most organizations still run high-stakes, repeatable processes manually — including recruitment, lead qualification, onboarding, and reporting. This agent demonstrates what becomes possible when those processes are designed with AI as a workflow partner.

🔁

Consistency at Scale

Every job is evaluated against the same weighted criteria, every time. No variation based on who ran the search or what day it was. The system applies the same judgment at scale that a skilled human would apply to a handful of listings.

⏱

Time Returned to High-Value Work

Manual job searching, deduplication, scoring, and document tailoring can consume hours per week. This agent collapses that to a Monday morning review of pre-scored, pre-drafted applications — freeing the human for decisions only a human can make.

🧠

Intelligence That Compounds

Every run adds to the outcome dataset. Over time, the team learns which sources produce interviews, which score ranges convert, and which keywords land. The system gets smarter as it operates — without requiring a developer to adjust it.

The broader principle: This pipeline is a proof of concept for any team that runs repetitive, judgment-based workflows. The same architecture — Config-driven parameters, modular scenarios, AI scoring, human review — can be applied to customer onboarding, content workflows, sales qualification, vendor evaluation, and more. The agent does not replace the team. It removes the mechanical work so the team can focus on the work that requires them.

System Architecture

Four Scenarios. One Closed Loop.

Each scenario has one job and fails independently. A broken data source does not take down scoring logic. All configuration lives in Airtable — change any parameter by editing one cell, no automation editing required.

01

AI Job Finder Scenario 1 — Controller

Reads all parameters from the Config table, splits keywords into an array, queries Tavily for job listings per keyword, and sends each result to Scenario 2. Fires a weekly completion email.

02

AI Job Finder Scenario 2 — Clean + Filter

Applies a series of automated filters and deduplicates each job against the existing Jobs table. Forwards only new, qualified listings to Scenario 3 for scoring.

03

AI Job Finder Scenario 3 — Score + Store

Claude researches the company and scores 7 dimensions. Make calculates the weighted formula — not the AI. A 4-way router stores each job under the correct score tier in Airtable.

04

AI Job Finder Scenario 4 — Doc Generator

Fetches high-scoring jobs, reads the correct base documents from Google Drive, sends to Claude for targeted injection, creates a customized output document, and emails the link.

This separation is what makes the agent maintainable by non-technical team members. No one needs to open Make.com to change how the system behaves.

Automated Filter Logic

Multiple filter conditions are applied automatically to each incoming job
Filters evaluate the job title, content, and source URL simultaneously
All filter conditions must pass — any single failure stops the job
Specific filter criteria are configurable and not publicly disclosed

Deduplication Logic

Every incoming job URL is checked against the full Jobs table
Match found → job is skipped. No match → job proceeds to scoring.
The same listing is never scored twice across multiple weekly runs
Deduplication happens before the Claude API call — saves tokens and cost

Scoring System

7-Dimension Weighted Formula

Document Selection

Why There Are Multiple Resume & Cover Letter Versions

The Philosophy Behind Multiple Versions

Every resume and cover letter in this system was written entirely by hand — not generated by AI, not templated, and not interchangeable. Each version reflects a specific industry context, a different set of values the employer is likely to hold, and a distinct aspect of my professional story that resonates most with that type of organization.

They are not changed by the automation. They do not get rewritten. They are preserved in full because the human voice, the personal narrative, and the philosophical framing are the most important things a document can carry — and those are the first things lost when AI takes over the writing.

Because every job that reaches document generation has already passed through the scoring system and scored extremely high, the keywords being injected are inherently relevant. High-scoring roles are already aligned to my skills and experience. Injecting those keywords into existing, genuinely written content is a targeted, honest optimization — not a fabrication.

The Injection Standard — What Claude Does and Does Not Change

The original resume and cover letter are preserved in full. Both documents are saved exactly as written before any changes are made. What Claude does is targeted and minimal: inject the company name, the specific job title, and relevant keywords where they fit naturally into the existing text — and optionally add one or two factual details about the company that strengthen what is already being said.

What Claude Injects Into the Resume

Job title — updated in the header to align with the target role
Summary — relevant keywords woven naturally into existing sentences
Up to 3 bullet points — only where a keyword fits without forcing it
If a keyword does not fit naturally — it is skipped entirely

What Claude Injects Into the Cover Letter

Company name — updated throughout wherever the hiring team is referenced
Job title — updated wherever the role is referenced in the text
Relevant keywords — injected naturally into existing sentences only
1–2 company facts — only if they strengthen something already being said

The candidate's own words are the most important element of every document. The philosophical framing, personal stories, voice, and narrative structure are never touched. What arrives in the output document should feel indistinguishable from something written personally for that company — because it was. The automation only handles the mechanical parts. The human wrote everything that matters.

Claude Output Format

===ATS KEYWORDS=== — top 10 keywords extracted from the JD
===RESUME=== — full original resume with minimal surgical injections
===COVER LETTER=== — full original cover letter with targeted updates only
===SUMMARY=== — top 3 alignment points for the reviewer's quick scan

Why This Only Runs for Top Scores

Document generation only triggers for Must Apply scores (≥ 8.5)
Every role at this threshold has already passed all filters and scoring
Keywords from the JD are inherently aligned to the candidate's skills
Injection is honest optimization — not fabrication — because fit is confirmed first
